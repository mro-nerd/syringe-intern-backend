# syringe-intern-backend

> **Backend Service** for the Syringe.ai Intern Assignment — an asynchronous task processing system built with **Node.js**, **Express**, **Upstash QStash**, and **MongoDB Atlas**.

---

## 📐 Architecture & Flow

The backend operates as an event-driven task processing system. It coordinates with MongoDB Atlas for persistent storage, and uses Upstash QStash as an external message broker to decouple HTTP requests from resource-heavy processing.

```
┌─────────────┐        ┌──────────────┐        ┌─────────────┐
│   React     │ POST   │   Express    │ Publish │   Upstash   │
│  Frontend   │───────▶│   Backend    │────────▶│   QStash    │
│  (Vite)     │        │              │         │  (Queue)    │
└──────┬──────┘        └──────┬───────┘         └──────┬──────┘
       │                      │                        │
       │  GET (poll)          │  Read/Write             │  Webhook
       │  /api/jobs/:id/status│                        │  POST /api/webhooks/qstash
       │                      ▼                        │
       │               ┌──────────────┐                │
       └──────────────▶│   MongoDB    │◀───────────────┘
                       │   Atlas      │   (via ngrok tunnel)
                       └──────────────┘
```

### Async Processing Flow:
1.  **Job Creation:** The client issues a `POST /api/jobs` request containing a `taskType`.
2.  **State Insertion:** The backend initializes a new job document in MongoDB with a state of `PENDING` and a progress of `0%`.
3.  **Publishing Message:** The backend publishes a message payload containing the `jobId` to Upstash QStash. The server immediately returns the new job details with HTTP Status `201` to the frontend.
4.  **Webhook Trigger:** QStash holds the message and fires a HTTP webhook (`POST /api/webhooks/qstash`) to the backend. Because the server is running locally, an **ngrok tunnel** is used to expose the local server port to the internet.
5.  **Signature Verification:** The incoming webhook request is passed through a custom security middleware (`verifyQStash.js`) which verifies the cryptographic signature generated by Upstash.
6.  **Simulated AI Task:** Once verified, a simulated background process starts. The progress of the task is updated in increments of 20% every 1.5 seconds. The database document is updated on each tick.
7.  **Client Status Retrieval:** In parallel, the frontend polls the `GET /api/jobs/:jobId/status` endpoint to track real-time progression.

---

## 🚀 Key Features

*   **Cryptographically Secure Webhooks:** Verifies HMAC SHA-256 signatures on incoming QStash requests using current and next keys to allow seamless key rotation.
*   **Decoupled Architecture:** Task queuing prevents server starvation; slow operations run in isolation from client requests.
*   **Audit logs (Status History):** Every job status transition (`PENDING` → `PROCESSING` → `COMPLETED`/`FAILED`) is appended to a `statusHistory` array with precise timestamps.
*   **Graceful Error Handling:** Catches database disconnects, invalid task requests, or unauthorized webhook calls.

## 🏁 Step-by-Step Installation & Run

### 1. Install Project Dependencies
```bash
git clone https://github.com/<your-username>/syringe-intern-backend.git
cd syringe-intern-backend
npm install
```

### 2. Set Up MongoDB Atlas
1.  Sign in to [MongoDB Atlas](https://www.mongodb.com/cloud/atlas/register).
2.  Deploy a free Shared M0 Cluster.
3.  Navigate to **Database Access** and create a user database profile with read/write access.
4.  Navigate to **Network Access** and select **Add IP Address** (choose `0.0.0.0/0` to allow access from any location for development, or specify your public IP).
5.  Click **Connect** on the Cluster panel and copy the "Node.js Driver" connection string. Add this to `.env` as `MONGODB_URI`.

### 3. Set Up Upstash QStash
1.  Register/login to [Upstash Console](https://console.upstash.com/qstash).
2.  Copy the token credentials from the dashboard:
    *   `QSTASH_TOKEN` (Rest API Client Token)
    *   `QSTASH_CURRENT_SIGNING_KEY`
    *   `QSTASH_NEXT_SIGNING_KEY`
3.  Add these keys to your `.env` file.

### 4. Create an ngrok Tunnel
Because Upstash QStash needs to make a `POST` request back to your computer, you must expose your local port (`4000`) using ngrok.
1.  Download and install [ngrok](https://ngrok.com/download).
2.  Authenticate your CLI agent: `ngrok config add-authtoken <your-authtoken>`
3.  Fire up the HTTP tunnel:
    ```bash
    ngrok http 4000
    ```
4.  Copy the public forwarding URL (e.g. `https://xxxx.ngrok-free.app`) and paste it as `WEBHOOK_BASE_URL` in `.env`.

### 5. Launch the Application
Run the backend server in development mode (with hot reloading enabled via nodemon):

```bash
npm run dev
```

If successful, you will see output like:
```text
🚀 syringe-intern-backend running on port 4000
✅ MongoDB connected: cluster-name.mongodb.net
```

## 🌐 API Reference

### Jobs API (`/api/jobs`)

#### 1. Create a Job
*   **Method:** `POST`
*   **Endpoint:** `/api/jobs`
*   **Request Body:**
    ```json
    {
      "taskType": "photo_generation"
    }
    ```
    *   *Supported task types:* `photo_generation`, `video_generation`, `ad_copy`, `target_audience`
*   **Response (`201 Created`):**
    ```json
    {
      "message": "Job created successfully and queued.",
      "job": {
        "jobId": "a823c914-7a0d-4001-a1e4-8cfd799015c9",
        "taskType": "photo_generation",
        "status": "PENDING",
        "progress": 0,
        "statusHistory": [
          { "status": "PENDING", "timestamp": "2026-07-01T18:00:00.000Z" }
        ],
        "_id": "64a0f..."
      }
    }
    ```

#### 2. Get Job Status (Polling Endpoint)
*   **Method:** `GET`
*   **Endpoint:** `/api/jobs/:jobId/status`
*   **Response (`200 OK`):**
    ```json
    {
      "jobId": "a823c914-7a0d-4001-a1e4-8cfd799015c9",
      "status": "PROCESSING",
      "progress": 40,
      "result": null,
      "error": null
    }
    ```

#### 3. List All Jobs
*   **Method:** `GET`
*   **Endpoint:** `/api/jobs`
*   **Response (`200 OK`):** Returns an array containing the history of all processed and active jobs.

## 🔒 Security Middleware (`verifyQStash.js`)

Webhooks are protected using a cryptographic signature verified inside [verifyQStash.js](file:///d:/Downloads/cyringe/syringe-intern-backend/src/middleware/verifyQStash.js).
*   On receipt of a request, the server reads the HTTP header `upstash-signature`.
*   It computes the HMAC SHA-256 hash using the incoming payload, timestamp, and configured signing keys (`QSTASH_CURRENT_SIGNING_KEY` and `QSTASH_NEXT_SIGNING_KEY`).
*   If the computed hash matches the signature, processing continues. Otherwise, the server responds with a `401 Unauthorized` block to protect your database and resources from external spam.
